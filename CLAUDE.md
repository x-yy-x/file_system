# CLAUDE.md — 文件系统管理项目

## 项目简介

操作系统课程项目：在内存中模拟虚拟磁盘，实现一个支持 CLI + GUI 的简易文件系统。  
技术栈：**Java**（Spring Boot 后端）+ **Vue 3**（前端）  
存储方案：索引分配 + 位图法

## 开发总要求

### 1. 功能覆盖

必须完整实现以下全部功能，不允许遗漏：

- **基础**：格式化、创建/删除/显示/切换目录、创建/打开/关闭/写/读/删除文件
- **存储**：索引分配（10 直接块 + 一级间接 + 二级间接）
- **空闲管理**：位图法
- **目录**：多级树形目录
- **持久化**：退出保存到 `filesystem.img`，启动自动恢复
- **附加操作**：rename / copy / move / stat / tree
- **两种界面**：Java 内置 CLI + Vue 图形界面（通过 REST API 通信）

### 2. 进度节点

- **截止日期**：6 月 24 日
- **分批提交**：每完成一个可运行的功能阶段，及时 commit 并 push

### 3. 版本控制

- **远程仓库**：`https://github.com/x-yy-x/file_system.git`
- **提交规范**：
  - 每次完成一个独立功能后立即 commit + push
  - 不要积攒多个功能一次性提交
  - commit message 用中文简述本次完成的内容，如 `feat: 实现位图分配与回收`、`feat: 添加 CLI 命令解析器`
  - 代码通过测试后再提交

## 代码要求

### 1. 高内聚低耦合

- **每个类只负责一个职责**，一个类不宜超过 300 行
- 模块间通过接口或抽象类依赖，不直接耦合具体实现
- 跨模块调用走定义好的 API 方法，不绕路访问内部数据结构
- 工具方法抽取为独立的 Utility 类，不散落在业务代码中

### 2. 分段清晰

- 类内部按以下顺序组织：常量 → 字段 → 构造方法 → 公开方法 → 私有方法
- 不同逻辑段之间空一行分隔
- 方法体不超过 50 行，超长则拆分为多个私有方法
- 方法命名自解释，一个方法只做一件事

### 3. 注释规范

- **写描述性注释，不写逐行注释**
- 类注释：说明该类职责（1-2 行）
- 公开方法注释：说明功能、参数、返回值、异常（JavaDoc 风格）
- 私有方法注释：只在逻辑复杂或非显而易见处加一行说明
- **禁止**：在函数体内每行都写注释、注释明显自解释的代码（如 `i++ // i 加 1`）、大段注释掉的死代码
- 接口/抽象方法必须有完整 JavaDoc

**示例（正确）：**

```java
/**
 * 在位图中分配一个空闲盘块。
 * @return 分配到的盘块号，磁盘已满时返回 -1
 */
public int allocateBlock() {
    for (int i = 0; i < blockCount; i++) {
        int byteIdx = i / 8;
        int bitIdx = i % 8;
        if ((bitmap[byteIdx] & (1 << bitIdx)) == 0) {
            bitmap[byteIdx] |= (1 << bitIdx);
            freeBlockCount--;
            return i;
        }
    }
    return -1;
}
```

**示例（错误——注释过度）：**

```java
public int allocateBlock() {
    // 遍历所有块
    for (int i = 0; i < blockCount; i++) {
        // 计算字节索引
        int byteIdx = i / 8;
        // 计算位索引
        int bitIdx = i % 8;
        // 检查该位是否为0（空闲）
        if ((bitmap[byteIdx] & (1 << bitIdx)) == 0) {
            // 将该位置为1（已分配）
            bitmap[byteIdx] |= (1 << bitIdx);
            // 减少空闲块计数
            freeBlockCount--;
            // 返回块号
            return i;
        }
    }
    // 没有空闲块
    return -1;
}
```

### 4. 命名规范

- 类名：大驼峰，名词，如 `BitmapManager`、`DiskDriver`
- 方法名：小驼峰，动词或动词短语，如 `allocateBlock()`、`readInode()`
- 常量：全大写 + 下划线，如 `BLOCK_SIZE`、`MAX_OPEN_FILES`
- 包名：全小写，如 `com.fs.model`、`com.fs.manager`
- 布尔方法用 `is`/`has`/`can` 开头，如 `isBlockFree()`

### 5. 异常处理

- 使用自定义异常类（如 `FileSystemException`）包装底层错误
- 不捕获异常后静默忽略，必须记录或向上抛出
- 业务逻辑错误（如文件不存在、磁盘已满）用返回值或业务异常表达，不用运行时异常

### 6. 测试

- 每个核心类都要有对应的单元测试（JUnit 5）
- 测试方法名用 `testXxx_shouldYyy_whenZzz` 格式
- 关键场景必须写集成测试（保存后加载验证、完整工作流）
- 提交前确保测试通过
